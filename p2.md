판정: 예 — 상태판정은 `세부 이벤트(event_status)`와 `보드 상태(board_state)`를 분리하는 2-Layer 구조가 가장 안전합니다.

근거: WAHA는 `message.any`로 모든 메시지 생성 이벤트를 받고, 그룹은 `@g.us`로 식별되며, `participant`와 `replyTo`를 함께 받아 그룹 발신자 식별과 thread 연결이 가능합니다. `message.reaction`도 별도 이벤트로 받을 수 있고, `session.status`로 세션 상태 모니터링이 가능합니다. ([WAHA][1])

다음행동: 아래 `상태 분류 규칙표 v0.1`과 `PostgreSQL 스키마 초안 v0.1` 그대로 시작하고, 실제 메시지 3~7일치를 붙여 false positive만 보정하면 됩니다.

## 1) 상태 분류 기준

### 1-1. 권장 상태 체계

| 세부 이벤트 (`event_status`) | 보드 상태 (`board_state`) | 의미                                   |
| ----------------------- | --------------------- | ------------------------------------ |
| `REQUEST`               | `NEW`                 | 요청 생성, 아직 착수 전                       |
| `ASSIGNED`              | `ASSIGNED`            | 담당자 지정됨                              |
| `ARRIVED`               | `IN_PROGRESS`         | 차량/화물/문서 도착                          |
| `STARTED`               | `IN_PROGRESS`         | loading/offloading/work 시작           |
| `ONGOING`               | `IN_PROGRESS`         | 진행 중, follow-up 중                    |
| `PARTIAL_DONE`          | `IN_PROGRESS`         | 일부 완료, balance 남음                    |
| `WAITING_GATE`          | `HOLD`                | gate/NOC/security/pass 대기            |
| `WAITING_DOC`           | `HOLD`                | BOE/DO/MSDS/FANR/TPI 등 문서 대기         |
| `WAITING_EQUIPMENT`     | `HOLD`                | crane/forklift/rigger/operator 대기    |
| `WAITING_REPLY`         | `HOLD`                | 회신/승인/확답 대기                          |
| `WAITING_WEATHER`       | `HOLD`                | high tide/fog/rain/wind/anchorage 대기 |
| `WAITING_APPROVAL`      | `HOLD`                | permit/approval/dispensation 대기      |
| `DONE`                  | `DONE`                | 완료                                   |
| `RELEASED`              | `DONE`                | trailer/container/asset release 완료   |
| `CANCELED`              | `CANCELED`            | 취소/연기/보류종결                           |

핵심은 **사용자 화면에는 `NEW / ASSIGNED / IN_PROGRESS / HOLD / DONE`만 노출**하고, 내부적으로는 `WAITING_DOC`, `WAITING_GATE` 같은 세부 원인을 유지하는 방식입니다.

## 1-3. 우선순위 규칙

이 부분이 중요합니다. 단어 사전만으로는 오분류가 생깁니다.

| 우선순위 | 규칙                                                          |
| ---- | ----------------------------------------------------------- |
| 1    | **구문(phrase)** 이 **단일 키워드**보다 우선                            |
| 2    | `DONE` + `balance/remaining/pending` 동시 존재 시 `DONE` 금지      |
| 3    | `ARRIVED` + `waiting forklift/crane/gate` 동시 존재 시 `HOLD` 우선 |
| 4    | `ongoing` + `resume 10:00`는 `IN_PROGRESS`                   |
| 5    | `partial delivery` + `balance qty`는 `PARTIAL_DONE`          |
| 6    | `noted only`는 상태 변경 금지                                      |
| 7    | 같은 thread에서 **가장 최근 명시 상태**가 이전 상태를 덮어씀                     |
| 8    | reaction으로 수동 오버라이드 허용 시, **수동값이 자동분류보다 우선**                |

## 2) DB 스키마 초안 v0.1

WAHA 쪽은 `message.any`, `message.reaction`, `replyTo`, 그룹 `@g.us`, `session.status`, webhook retries, HMAC 검증을 바로 쓸 수 있게 raw event를 남기는 구조가 좋습니다. PostgreSQL 쪽은 raw payload와 추출 결과를 `jsonb`로 보관하고, dashboard가 자주 보는 `OPEN/HOLD` slice는 partial index로 분리하는 방식이 적합합니다. PostgreSQL 공식 문서상 `jsonb`는 containment/existence 연산을 지원하고, partial index는 특정 조건의 자주 조회되는 부분집합에 유리하지만 predicate 일치 조건을 신중히 맞춰야 합니다. ([WAHA][1])

### 2-1. 설계 개요

| 계층   | 테이블                                                   |
| ---- | ----------------------------------------------------- |
| 수집   | `wa_raw_event`, `wa_message`, `wa_message_reaction`   |
| 기준정보 | `ops_group`, `ops_contact`                            |
| 파싱   | `wa_message_parse`, `classification_rule`             |
| 업무카드 | `work_item`, `work_item_message`                      |
| 이력   | `work_item_status_history`, `work_item_owner_history` |

## 그대로 써도 되는 부분

이 부분은 유지해도 됩니다.

| 항목                                                            | 판정 | 이유                       |
| ------------------------------------------------------------- | -- | ------------------------ |
| `board_state` / `event_status` 2단 구조                          | 유지 | 화면은 단순, 내부는 상세 추적 가능     |
| `wa_raw_event → wa_message → wa_message_parse → work_item` 계층 | 유지 | 수집/파싱/카드 분리 구조가 맞음       |

| 키워드 사전 + precedence 규칙                                        | 유지 | 운영 메시지 자동분류의 최소 안전장치     |

## 그대로 쓰면 안 되는 부분

여기는 수정이 필요합니다.

### 1) raw/parse 테이블은 `public`에 두지 마십시오

현재 초안은 모든 테이블을 같은 레벨에서 쓰는 구조인데, Supabase는 `public` schema가 Data API에 기본 노출됩니다. 민감한 raw 메시지·parser 결과는 `private` 계열 schema로 분리하는 것이 맞습니다. Supabase도 private schema 사용을 권장합니다. ([Supabase][1])

권장:

* `ops_private.wa_raw_event`
* `ops_private.wa_message`
* `ops_private.wa_message_parse`
* `ops_private.classification_rule`

남길 것:

* `public.work_item`
* `public.work_item_status_history`
* `public.work_item_owner_history`

### 2) RLS 없이는 그대로 쓰지 마십시오

Supabase는 Data API에 노출되는 테이블에 RLS를 요구하고, `service_role`은 RLS를 우회하므로 브라우저에 두면 안 됩니다. 즉, 현재 스키마는 **구조는 맞지만 보안 설정 전제**가 빠져 있습니다. ([Supabase][1])

운영 원칙:

* 브라우저: `anon` 키 + RLS
* Vercel 서버: `service_role` 또는 secret key
* 브라우저에 `service_role` 절대 금지

### 3) Realtime은 `work_item` 계층에만 거십시오

현재 초안대로 raw/message/parse까지 실시간으로 열면 과합니다. Supabase Postgres Changes는 publication에 테이블을 추가해야 하고, docs에서도 규모가 커지면 Broadcast를 더 권장합니다. 그래서 MVP는 `public.work_item`, `public.work_item_status_history`만 Realtime 대상이 맞습니다. ([Supabase][2])

### 4) Vercel Cron 전제는 플랜 따라 다릅니다

문서상 후속 요약/재분류/stale-check를 자주 돌리려면 Hobby로는 부족할 수 있습니다. Vercel Cron은 Hobby가 하루 1회, Pro가 분 단위입니다. 운영타워면 사실상 Pro 전제가 더 맞습니다. ([Vercel][3])

### 5) 상태 사전은 현재 그대로보다 “현장형 키워드”를 더 넣어야 합니다

지금 사전은 generic합니다. 하지만 당신 내부 로그 패턴상 아래 표현은 꼭 추가해야 정확도가 올라갑니다.

* SHU 계열: `partial delivery`, `hold at DSV`, `deliver to MOSB`, `Arrived`, `Unloading start time`, `Unloading complete time`, `balance qty / issue` 
* JPT71 계열: `high tide`, `berth congestion`, `anchorage standby`, `waiting manifest`, `manifest owner`, `FW/MGO`, `pre-arrival` 

즉, **현재 사전은 base**, **그룹별 사전이 추가로 필요**합니다.
