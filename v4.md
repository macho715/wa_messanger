# Message Work Board Spec — Merged Final v0.2

## 0) 최종 결론

- **상태모델:** `event_status` + `board_state` 2-Layer 유지
- **핵심 객체:** `work_item`(업무카드)
- **수집 파이프라인:** `WAHA → Vercel webhook → Supabase`
- **보안 원칙:** raw/parse/rule은 `ops_private`, board 계층만 `public`
- **Realtime 범위:** `public.work_item`, `public.work_item_status_history`만
- **브라우저 권한:** read only
- **서버 권한:** Vercel server only write/upsert

---

## 1) 2-Layer 상태 모델

### 1-1. 사용자 노출 상태 (`board_state`)

- `NEW`
- `ASSIGNED`
- `IN_PROGRESS`
- `HOLD`
- `DONE`
- `CANCELED`

### 1-2. 내부 세부 상태 (`event_status`)

| event_status | board_state | 의미 |
|---|---|---|
| REQUEST | NEW | 신규 요청 |
| ASSIGNED | ASSIGNED | 담당자 지정 |
| ARRIVED | IN_PROGRESS | 차량/화물/문서 도착 |
| STARTED | IN_PROGRESS | 작업 시작 |
| ONGOING | IN_PROGRESS | 진행 중 |
| PARTIAL_DONE | IN_PROGRESS | 일부 완료, 잔량 존재 |
| WAITING_GATE | HOLD | gate/NOC/security/pass 대기 |
| WAITING_DOC | HOLD | BOE/DO/MSDS/FANR/TPI 등 문서 대기 |
| WAITING_EQUIPMENT | HOLD | crane/forklift/operator 대기 |
| WAITING_REPLY | HOLD | 회신/확답 대기 |
| WAITING_WEATHER | HOLD | high tide/fog/rain/wind/anchorage 대기 |
| WAITING_APPROVAL | HOLD | permit/approval/dispensation 대기 |
| DONE | DONE | 완료 |
| RELEASED | DONE | release/return/gate out 완료 |
| CANCELED | CANCELED | 취소/연기/보류종결 |

원칙:
- 화면은 `board_state` 중심으로 단순화
- 내부 로직/상세 패널/분석은 `event_status` 유지
- `HOLD`는 반드시 이유코드 유지

### 2-2. precedence

1. **phrase match > 단일 keyword**
2. `DONE + balance/remaining/pending` 동시 존재 시 완료 금지
3. `ARRIVED + waiting forklift/crane/gate` 동시 존재 시 `HOLD` 우선
4. `partial delivery + balance qty`는 `PARTIAL_DONE`
5. `noted only`는 상태 변경 금지
6. 같은 thread에서는 **가장 최근 명시 상태**가 이전 상태를 덮어씀
7. reaction 기반 수동값이 있으면 자동분류보다 우선

## 5) 보안/운영 원칙

### 5-1. RLS

- `public.work_item*`는 RLS 활성화
- 브라우저는 `anon/authenticated` read only
- insert/update/delete는 브라우저 금지
- Vercel server만 `service_role`로 write
- `service_role`은 절대 브라우저에 노출 금지

### 5-2. Realtime

MVP 대상:
- `public.work_item`
- `public.work_item_status_history`

비대상:
- `wa_raw_event`
- `wa_message`
- `wa_message_parse`

### 5-3. Cron

- Hobby: 운영형 stale-check 부족 가능
- Pro: 실시간 운영에 적합
- 초기 기능은 webhook-driven 우선
- cron은 `stale-check`, `daily-summary`, `rebuild` 보조용

## 7) 최소 구현 순서

1. `WAHA → Vercel /api/webhooks/waha`
2. `ops_private.wa_raw_event` 저장
3. `ops_private.wa_message` 정규화 저장
4. `ops_private.wa_message_parse` 상태/담당자/업무유형 추출
5. `dedupe_key` 기반 `public.work_item` upsert
6. 상태 변경 시 `public.work_item_status_history` 기록
7. 대시보드는 `board_state` 중심 렌더링
8. 우측 상세창에 `event_status`, `hold_reason`, 근거 메시지 표시

---

## 8) 그대로 유지할 부분

- `board_state` / `event_status` 2-Layer
- `wa_raw_event → wa_message → wa_message_parse → work_item`
- `work_item_status_history`
- `work_item_owner_history`
- `dedupe_key` 기반 upsert
- 키워드 사전 + precedence 규칙

---

## 9) 반드시 패치할 부분

1. raw/parse/rule 테이블을 `public`에 두지 않음
2. `public.work_item*`에만 RLS + Realtime 적용
3. 브라우저 read only, 서버 write only
4. `service_role`은 Vercel 서버 전용
5. generic dictionary만 쓰지 말고 그룹별 사전 추가
