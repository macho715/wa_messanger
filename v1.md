판정: **예 — 둘 다 바로 드립니다. `최종 폴더 구조`는 이대로 시작하면 되고, `patched SQL`은 `ops_private + public` 2-schema로 가면 됩니다.**

근거: 현재 기준은 제품을 `Shipment graph + message work-card layer`로 나누고, 메시지는 evidence로만 다루며, full journey와 `HOLD`를 유지해야 합니다. 또한 P2 분리, 승인형 실행, 감사 로그를 기본 계약으로 둬야 합니다.     

다음행동: **1) 아래 폴더 구조로 정리, 2) SQL 1회 적용, 3) `app/api/waha/webhook/route.ts`부터 연결** 순서로 진행하면 됩니다.

## 3) Supabase용 patched SQL

가정:

* Supabase Postgres 사용
* 브라우저는 direct write 금지
* write는 Vercel server route만 수행
* `ops_private`는 raw/message/parse/rule 저장소
* `public`은 UI 소비용 최소 테이블만 유지

## 5) 운영 메모

* `public`에는 **UI가 바로 읽어야 하는 최소 결과만** 둡니다.
* `ops_private`에는 raw payload, parser 결과, 연락처, rule 사전 등 민감/중간 데이터를 둡니다.
* `HOLD`는 반드시 1급 visible column으로 유지해야 합니다. 
* `Shipment` current stage/location과 `WorkCard` current status/owner는 분리해야 합니다. 메시지가 곧 shipment truth가 되면 안 됩니다. 
* P2 자료는 DB/문서/프롬프트 모두에서 별도 권한과 분리 저장 원칙을 유지해야 합니다. 
* 승인형 실행과 audit trace는 UI/백엔드 계약으로 남겨야 합니다.  

원하면 다음 답변에서 바로 `app/api/waha/webhook/route.ts`와 `lib/waha/parser.ts` 실전 코드까지 이어서 드리겠습니다.
